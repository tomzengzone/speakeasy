# PR-001: PR1 精简流程决策与迁移安全合同

## Objective

建立“开发流程精简治理”的正式决策和迁移安全边界：说明为何删除强制 Spec/AC 能缩短 AI coding 上下文与反馈链，定义产品事实、工程 Contract、TC oracle 和交付状态的唯一 ownership，并通过 Spec/AC retirement manifest 防止精简过程中丢失 116/262 legacy 事实；PR1 本身不提前改变运行中的开发流程。

## Included Scope

- ADR 0007 的流程现状分析、AI 辅助 coding 成本、成熟团队实践、目标最小上下文、事实/交付双轴、风险分级、PR1–PR4、回滚和延迟 supersede 语义。
- 对现有强制 `FR -> Spec -> AC -> TC` 文档跳数、重复事实、Gate 等待和上下文负担建立可供 PR2 对比的基线口径。
- 116 个 `docs/product/increments/**/*.md` 路径快照及其中 262 个 Spec/AC 定义 ID 的逐条 migration-only 清单。
- 每条记录的 `old_path`、`old_id`、`source_kind`、`status`、`destinations`、中文 reason 和 owner；初始条目不得伪造已迁移去向。
- Manifest Schema、专用 validator、正负例测试、4 个 migration-only Artifact 注册和治理回归基线。
- 批准后重新执行全部定向验证和独立只读治理审查；同范围缺陷允许在本 revision 内修正。

## Excluded Scope

- 不修改 workflow、Definition of Done、现有 Gate/Actor/Exchange/Intent 或现有 Skill 的交付语义。
- 不修改 Product Base、Story Map、Capability Registry、Stage、Increment 或应用代码、应用测试、API、数据库、AI runtime 和部署配置。
- 不迁移、重新批准或发明产品行为；不把 manifest 作为任何下游 Artifact 的直接或条件输入。
- 不修改 ADR 0001 历史内容，不激活新 FR/TC authority，不提交或发布远程 PR。

## Acceptance Criteria

- ADR 明确任务目标是精简开发流程，User Story/Vertical Slice 是产品输入模型而非治理目标；状态为 migration preparation，只有 PR-003 原子激活后才实际 supersede ADR 0001。
- 目标流程明确区分：Story/VS/FR 的产品事实、API/Domain/Persistence/AI/UX Contract 的工程事实、TC 的持久 oracle、Issue/PR/CI/rollout 的一次性交付状态。
- 明确 AI coding 最小上下文包、小批量 PR、test-first、快速本地反馈、exact-commit CI 和风险触发治理；删除 Spec/AC 不得删除异常、oracle、Contract 或验收责任。
- 决策与计划明确给出文件级落地顺序、目标 Definition of Done 和量化最终验收指标；PR1 只建立合同，不把目标 DoD 提前激活。
- 路径数、规范化路径 SHA-256、定义 ID 集合和 manifest `(old_path, old_id)` 集合精确一致；标题和明确 definition/coverage 表首列可识别，正文引用和普通表格不会误算。
- 262 条初始记录均为 `grandfathered-unverified`、空 `destinations`、中文 reason 和明确 owner；Schema 支持一个旧对象对应多个 typed destination，且包含 VS、FR、API/Domain/Persistence/AI/UX Contract 和 executable regression TC。
- 4 个迁移 Artifact 唯一注册，owner/lifecycle/validation command 正确；manifest 不出现在任何 Artifact 的 `required_direct_inputs` 或 `conditional_inputs`。
- `G-SPEC`、`G-AC-TC`、workflow、DoD 和现有 Skills 保持不变并继续唯一生效。
- 所有定向 validator/test 和独立只读 `product_object_governance_check` 通过；用户看到证据后显式验收。
