# PR-003: PR3 前瞻式开发治理原子切换

## Objective

在一个可回滚的候选 commit 中启用精简后的正式开发流程：删除强制 Feature Spec、Acceptance Criteria 及其 Gate，把产品事实链收敛为 `Capability/Sub-capability classification -> User Story -> Vertical Slice -> mandatory Functional Requirement`，并建立 FR、Engineering Contract、VS 三类分层 Test Case。Story Map 持有 Story 到 Capability 与 VS 到 Story 的直接关系，FR 只直接引用 approved VS；FR-TC、Contract-TC、VS-TC 分别只直接引用 FR、Engineering Contract、VS。Canonical traceability 只从这些 owning sources 派生完整分支图与 coverage join，不独立定义或覆盖直接边。旧 Product Base / Increment 文档原地保留但整体退出 active authority，不执行内容迁移、逐条事实分类或无损继承证明。切换同时建立唯一 authority graph：Governance Contract 独占 Artifact/Gate 治理事实，Agent、Skill、Workflow 和 Template 仅保留各自职责。治理执行只依赖 Agent/native permission、最终 diff validator、CI 和独立审查，不建立 project-local Hook、runtime resolver 或 ephemeral bundle。候选 SHA 经完整 CI 与独立审查后，只有在用户单独授权下被 fast-forward 到受保护的 `refs/heads/speakeasy-20260705`，才成为机器可判定的 active baseline。

## Included Scope

- 更新 ADR 0007、workflow、Definition of Done、Artifact/Gate/Actor/Exchange/Intent、相关 Skills/Agents、validator、tests 和 CI required checks。
- 对仍由 active Governance/Engineering Artifact route 引用的 cross-cutting、SWC、release、Architecture、Domain、AI、OpenAPI 与 UX 文档只做 lineage/provenance 收敛：移除 Product Base、Increment、Spec/AC、Increment SWC Allocation 作为现行权威上游的表述，不改写工程行为、schema、边界或已接受实现事实。
- Candidate authority graph 必须从 index route、registered native Agent/config、active method Skill 及其直接链接 resource 派生；不得继续把整个 `.codex`、`.agents/skills` 或 `codex/templates` 目录一概视为 active baseline。`.codex/task-plans/**`、未登记 template、migration-only 文件和旧产品文档只作为历史/执行状态，不进入候选内容基线或默认上下文。
- 对 ADR 0007 建立定向 current-decision contract：必须明确 mandatory FR、FR/Contract/VS 三类 TC 及其唯一直接上游、derived canonical traceability、PR-002 superseded、forward-only/no-migration、旧文档 historical-only、Hook 全套方案已删除，以及 `refs/heads/speakeasy-20260705` exact-SHA baseline 激活模型；optional FR、shadow/262 迁移、active Spec/AC Gate、Hook/resolver/bundle 或候选内 `status=active` 只能作为明确 superseded/negative 说明，不能继续构成当前决策。
- 对 Story Map 建立定向 active-source contract：它是 User Story 与 Child VS 的唯一产品事实文件，Story 到 Capability、Child VS 到 Story 为唯一直接上游；`docs/product/user_stories.md`、Product Base、Increment Requirements/Spec/Acceptance/Test/Traceability 不得继续作为 source、prerequisite、fallback 或 authoritative link。Stage/Increment 只可作为明确 planning-only 元数据，Story Map 不复制 FR/TC/Contract 内容，也不得恢复 Spec/AC 字段或交付链。
- 在 `policy.json` 中固化职责分层：Governance Contract 独占 canonical path、accountable owner、contributor scope、lifecycle、Artifact direct/conditional inputs 和 Gate routing；Agent 只定义角色/权限/专业边界/handoff；Skill 只定义适用任务、方法步骤、内容规则和验证方法；Workflow 只定义阶段顺序、决策点和按 ID 解析 contract 的规则；Template 只定义内容字段与版式。非 owning layer 可以通过 Artifact/Gate ID 使用 contract，也可以在确有执行必要时保留与 contract 一致且由 validator 校验的 derived operational pointer，但不得把路径、owner、lifecycle、依赖或 Gate 重新声明为自己的 authority。
- 由 cutover validator 从 `index.json` 的候选 route set 构建 candidate authority graph；受保护的 `refs/heads/speakeasy-20260705` 是 accepted active baseline 的唯一机器可读 anchor，同一 exact SHA 在独立 candidate branch 只表示 candidate，在用户授权后 fast-forward 成为 baseline HEAD 才表示 active。候选与 active 使用同一套完整性、职责去重和 legacy 隔离检查，候选内容不携带会被误读为已接受基线的全局 `status=active`。审查范围包含 `AGENTS.md`、workflow、DoD、Skill Quality Standard、全部 active Agent/Skill/Skill resource、Gate/Exchange/Intent、validator 和 CI 入口；旧产品文档及已退出 route 的 PR-001 migration-only 文件只作为 historical evidence，不进入 active graph、默认上下文或新增 authoritative input。validator/test 中用于证明退役的 denylist literal 属于负向断言，不计作 active authority。
- Governance Contract 继续作为 canonical path、owner/contributor、Artifact inputs、lifecycle 与 Gate routing 的唯一 authority；Agent 配置和 native sandbox 负责角色与实际写权限，Workflow/Skill 按 Artifact/Gate ID 解析适用 contract。PR-003 不建立 project-local Hook、runtime governance resolver、ephemeral bundle、首次写入 deny/retry 或 actor/session/turn 绑定机制。
- 普通 validator 与 CI 对最终候选 diff 检查 governed Artifact scope、legacy authority 隔离、mandatory FR、三层 TC、canonical traceability、Engineering Contract 上游和 applicable Gate；独立 checker 继续验证产品对象和架构边界。`check_governance_write_scope.py` 最多作为可选本地预检，不拦截首次写入、不产生权限或 PASS bundle，也不能替代最终 CI 与独立审查。
- 建立 `FUNCTIONAL_REQUIREMENT_CATALOG`、`TEST_CASE_CATALOG` 和 canonical traceability 路由；Story Map 继续同时持有 User Story 与嵌套 Child Vertical Slice 的产品事实和 PM approval。

- 每个进入实现的 approved VS 必须至少关联一条 approved FR；每条 FR 必须只以 `source_vs_ids` 直接引用一个或多个 approved VS，不直接重复 Story ID 或把 Capability classification 当作第二条 lineage。FR 仍归属一个 Primary Capability/Sub-capability 并记录 affected Capability/Sub-capability，但这些字段只用于分类、编号与影响筛选。
- FR 按 Primary Capability/Sub-capability 分区和编号；大模块先拆稳定一级子功能，每条 FR 只表达一个原子、独立可测试的规则、不变量、边界或失败条件，不承载 Stage、Increment、PR、实现或执行状态。
- Canonical traceability 是由 owning sources 重建和校验的只读完整链路投影：从 Story Map 取得 `Capability/Sub-capability -> Story -> VS`，从 FR Catalog 取得 `VS -> FR`，从 Engineering Contract 取得 `FR -> affected Contract`，从 TC Catalog 取得 `FR -> FR-TC`、`Contract -> Contract-TC`、`VS -> VS-TC`，再连接 selector/evidence。它不得独立创建、修改或覆盖任何直接边；不一致时必须修复 owning source 后重新生成投影。
- 建立三类分层测试设计并固定唯一 direct-upstream edge：FR-TC 只记录 `source_fr_id`，Contract-TC 只记录 `source_contract_id`，VS-TC 只记录 `source_vs_id`。每条 FR 必须有最低成本的快速 FR-TC 或有 owner、原因和期限明确的例外；Contract 事实变化必须有 contract/integration/migration/AI-eval 等 Contract-TC；每个实施中的 VS 必须有用户可感知 integration/E2E VS-TC。VS-TC 对全部适用 FR 的覆盖由 canonical traceability 根据 VS 与 FR 直接边派生并校验，不在 VS-TC 中重复维护 FR ID 集合。
- FR-TC 优先在 unit/domain/service/contract/integration/widget 层快速发现模块内部缺陷；VS-TC 验证受影响数据库、后端、API、前端和用户界面的完整闭环，但不要求无关层参与。VS 完成前必须通过其定向全链路测试，不等待最终 release E2E 才发现问题。
- Test Case Catalog 保存稳定 oracle、Given/When/Then、边界/负例、测试层、scope、selector、测试脚本路径和执行命令，不保存 `passed/failed` 等易过期执行状态；执行结果只由绑定 exact commit SHA 的测试/CI 系统保存。
- `Engineering Contract` 仅作为 Architecture、API、Domain、Persistence、AI、UX 等现有工程 Artifact 的分类，不创建统一聚合 Contract、不合并或转移现有 accountable owner；各 Artifact 的上游只在 Governance Contract 中重接到适用 FR/其他 Engineering Artifact，并通过 traceability 解析 owning VS/Story，相关 Skill 不复制该依赖图。
- 完整执行 legacy Artifact disposition：退役 `PRODUCT_BASE_REQUIREMENTS`、`PRODUCT_BASE_SPEC`、`PRODUCT_BASE_ACCEPTANCE`、`PRODUCT_BASE_TRACEABILITY`、`PRODUCT_BASELINE`、`INCREMENT_REQUIREMENTS`、`INCREMENT_SPEC`、`INCREMENT_ACCEPTANCE`、`INCREMENT_TEST_CASES`、`INCREMENT_TRACEABILITY`、`INCREMENT_SWC_ALLOCATION` 的 active route；`STAGE_SCOPE` 与 `INCREMENT_DEFINITION` 仅保留规划、优先级和批次语义，禁止成为 FR、TC 或 Engineering Contract 的上游。

## Acceptance Criteria

- Active 产品事实链唯一为 `Capability/Sub-capability classification -> approved User Story -> approved Vertical Slice -> mandatory FR`；Capability 只分类和划边界，不定义行为。
- ADR 0007 的 current decision 与 revision 11 完全一致：FR 明确 mandatory；三类 TC 及 direct-edge matrix、derived traceability、forward-only/no-migration、PR-002 superseded、legacy historical-only、无 Hook/runtime resolver/bundle、exact-SHA baseline ref 均为正向现行规则。ADR 中不存在仍可被解释为现行义务的 optional FR、shadow/262 迁移、Spec/AC Gate、Hook preflight 或候选内 `status=active`。
- Story Map 持有 Story 到 Capability 与 Child VS 到 Story 的直接关系；FR 只以 `source_vs_ids` 记录 VS lineage，不重复 Story ID，也不把 Primary/Affected Capability/Sub-capability classification 当作第二条 lineage；canonical traceability 能无歧义解析完整 Story lineage。
- Story Map 明确自身是 User Story 与 Child VS 的唯一 canonical product source；对 `docs/product/user_stories.md`、Product Base、Increment Requirements/Spec/Acceptance/Test/Traceability 的正向 source/prerequisite/fallback/authoritative reference 数量为零。Stage/Increment 若出现必须明确 planning-only，Story Map 中 Spec/AC 字段、Gate 或交付步骤数量为零，FR/TC/Contract 正文复制数量为零。
- 每个进入实现的 approved VS 至少具有一条 approved FR；每条 FR 归属一个 Primary Sub-capability、表达单一独立可测试规则，并可关联一个或多个 approved VS。
- `G-SPEC`、`G-AC-TC` 和强制 Spec/AC Artifact/Skill 已退役；新流程不允许通过临时 Spec/AC 绕过 mandatory FR 或分层 TC。
- `product-spec-authority`、`acceptance-authority`、`feature-spec-generate`、`acceptance-criteria-generate` 不再出现在 active actor、method skill、intent/exchange、Gate 或 Skill discovery route 中；退役 Skill 不以 tombstone/redirect 形式继续可调用。
- 每条 FR 均映射至少一个稳定 FR-TC，或存在 owner、原因、影响和失效期限明确的例外；FR-TC 优先使用能证明规则的最低成本层级并可按 Capability/Sub-capability 定向运行。
- 每个实施中的 VS 均映射至少一个用户可感知的 integration/E2E VS-TC；VS-TC 只直接记录 `source_vs_id`，覆盖实际受影响的数据库、后端、API、前端和 UI 全链路及关键失败/降级路径。该 VS 的全部适用 FR coverage 由 canonical traceability 派生并验证，不在 VS-TC 中重复保存 FR ID 集合。
- API/Domain/Persistence/AI/UX Contract 发生事实变化时，对应 Contract-TC 已存在并通过适用 contract/integration/migration/AI-eval 验证；低风险不能豁免事实同步。
