# PR-004: PR4 真实功能试点与效率质量闭环

## Objective

选择一个 PM 已批准的真实小功能，在 PR-003 激活的精简流程下完成从产品事实到 exact-commit CI 的交付，测量文档触点、事实重复、AI context bundle、反馈轮次和质量证据，证明流程更短且行为精度、Contract 完整性和风险控制不降级；approved VS 是试点输入，不是试点目标。

## Included Scope

- PM 选择并确认一个边界足够小、可独立验收的 approved VS；若触发条件、前置状态、用户选择、状态变化、可见结果或关键异常不完整，先回到 owning VS 修正。
- 在实现前记录 legacy 基线与精简流程的 Artifact/Gate 触点、AI 输入引用集合和预计反馈节点；完成后记录实际值，不使用主观“更快”代替证据。
- 识别仅在多 Slice 复用的稳定规则并引用 canonical FR；单 Slice 规则保留在 VS。
- 在 canonical TC catalog 中先建立自包含回归 oracle 和稳定 selector，运行测试先失败，再进行最小实现并通过。
- 只更新实际受影响的 API/Domain/Persistence/AI/UX Contract；额外治理与发布控制按真实风险触发。
- 记录本地与 exact-commit CI 证据、独立 review 结论、产品事实缺口或治理缺口的回流位置，并完成任务级 Overall Evidence 候选。

## Excluded Scope

- revision 3 不选择具体 VS，不授权任何应用代码、应用测试、Contract 或产品事实路径。
- 不创建 Feature Spec、Acceptance Criteria 或 Increment authority 作为临时绕行层。
- 不把试点发现的产品事实缺口留给 TC/代码猜测，不把流程缺口伪装成产品规则。
- 不自动扩展第二个 Slice，不提交、推送或创建远程 PR。

## Allowed Paths

revision 3 无应用实现路径授权。PR-003 完成后，必须由 PM 提供 approved VS ID，随后运行 `revise-pr` 并将本节替换为以下精确路径集合后才可审批：owning Story/VS、适用 FR、canonical TC、受影响 Engineering Contract、最小应用代码、对应自动测试和必要 CI/证据路径。不得使用通配的整个应用目录。

## Acceptance Criteria

- 卡片在批准前记录明确的 approved VS ID、产品 owner、完整用户闭环、适用 FR/Contract、精确允许路径和风险分类；revision 与批准一致。
- 实际交付新增 Spec/AC 文件数为 0，新增 legacy authority 引用数为 0；同一行为规则只有一个 canonical owning source。
- AI coding context bundle 仅含 selected VS、适用 FR/Contract、相邻代码/测试和验证命令，并记录相对 legacy 强制链路减少的文档/Gate 触点。
- 总计划 Final Acceptance Metrics 在试点范围内全部有机器可读或可复核证据：Spec/AC 与 legacy authority 引用为 0、单一 owning source、TC/Contract 覆盖 100%、context bundle 不高于对照的 60%、自动验证与 exact-commit CI 通过率 100%、blocker 为 0。
- 测试设计先于实现，TC 自包含 Given/When/Then、边界、层级和 selector；保存可归因的先失败、后通过证据。
- 实现只消费 approved VS、可选 FR、canonical TC 和受影响 Contract，不读取 archived Spec/AC/Increment authority。
- 产品事实缺口回写 owning VS/FR 并重新批准；治理/Schema 缺口修正 owning governance Contract；不得创建临时 Spec/AC。
- 所有定向测试、contract/integration/eval、性能/安全/数据检查按影响通过，CI 结果绑定同一 exact commit。
- 产品、代码质量及适用架构 checker 独立通过；用户验收试点后 PR-004 才完成。
- 四个 PR 全部完成后汇总 Overall Evidence，由用户单独验收整个任务。
